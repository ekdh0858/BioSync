# PRD - BioSync

## 1. 제품 개요

BioSync는 모바일 앱과 웨어러블 기기에서 수집한 생체신호 데이터를 저장, 분석, 조회하는 플랫폼이다. 사용자는 일별 건강 지표와 이상징후를 확인할 수 있고, 관리자는 운영 화면에서 사용자 상태와 데이터 수집 현황을 모니터링할 수 있다.

## 2. 문제 정의

기존 건강 데이터 서비스는 다음 문제를 자주 가진다.

- 원본 생체신호와 분석 결과가 분리되지 않아 관리가 어렵다.
- 사용자용 기능과 관리자용 기능이 한 구조에 섞여 확장성이 떨어진다.
- 경고 이벤트 기준이 불분명해 운영자가 빠르게 대응하기 어렵다.

BioSync는 수집, 분석, 조회, 운영 모니터링을 분리된 구조로 설계해 이 문제를 해결한다.

## 3. 제품 목표

- Spring Boot 기반의 안정적인 백엔드 구조 구축
- 생체신호 수집 및 일별 요약 기능 제공
- 이상징후 탐지와 관리자 조회 기능 제공
- 사용자 영역과 관리자 영역의 권한 및 API 경로 분리
- 확장 가능한 데이터 모델과 패키지 구조 유지

## 4. 사용자 유형

### USER

모바일 앱을 사용하는 일반 사용자다.

주요 요구사항:

- 회원가입과 로그인
- 본인 디바이스 등록
- 생체신호 업로드
- 일별 건강 데이터 조회
- 이상징후 확인

### ADMIN

운영 관리를 담당하는 관리자다.

주요 요구사항:

- 사용자 목록 조회
- 특정 사용자 데이터 조회
- 이상징후 조회
- 시스템 수집 현황 확인

## 5. 핵심 시나리오

### 사용자 시나리오

1. 사용자는 앱에서 회원가입 후 로그인한다.
2. 웨어러블 디바이스를 본인 계정에 연결한다.
3. 앱은 주기적으로 생체신호 데이터를 배치 업로드한다.
4. 사용자는 앱에서 일별 요약 데이터와 이상징후를 확인한다.

### 관리자 시나리오

1. 관리자는 관리자 콘솔에 로그인한다.
2. 사용자의 업로드 현황과 Alert 발생 내역을 조회한다.
3. 비정상 수치가 반복되는 사용자를 확인한다.

## 6. 기능 요구사항

### 6.1 인증

- 회원가입
- 로그인
- JWT Access Token 기반 인증
- 인증이 필요한 API 접근 제어

### 6.2 디바이스 관리

- 디바이스 등록
- 사용자별 디바이스 조회
- 중복 등록 방지

### 6.3 생체신호 업로드

지원 데이터:

- 심박수
- 걸음 수
- 수면 시간
- 스트레스 지수

업로드 방식:

- Batch Upload

필수 조건:

- 요청 단위 유효성 검증
- 저장 실패 시 명확한 에러 반환
- 업로드 로그 기록

### 6.4 데이터 분석

일별 Summary에 포함할 항목:

- 평균 심박수
- 최대 심박수
- 최소 심박수
- 총 걸음 수
- 총 수면 시간

### 6.5 이상징후 탐지

예시 규칙:

- 심박수 120 초과
- 하루 걸음 수 급감
- 일정 기간 데이터 업로드 누락

### 6.6 관리자 기능

- 사용자 조회
- 특정 사용자 생체신호 조회
- Alert 조회
- 데이터 수집 현황 조회

## 7. 비기능 요구사항

- API 응답은 공통 포맷을 사용한다.
- Controller와 Service, Repository 책임을 분리한다.
- Entity와 API 응답 DTO를 분리한다.
- 주요 사용자 액션과 시스템 에러는 로그를 남긴다.
- 주요 비즈니스 로직은 Service 레이어 테스트를 작성한다.

## 8. 성공 지표

- 사용자가 일별 건강 지표를 안정적으로 조회할 수 있다.
- 관리자 화면에서 이상징후와 수집 현황을 빠르게 확인할 수 있다.
- 핵심 서비스 로직이 테스트로 보호된다.
- 문서와 실제 API 구조가 일치한다.

## 9. MVP 범위 제외

- Kafka 기반 이벤트 파이프라인
- MSA 전환
- 실시간 스트리밍 처리
- 머신러닝 기반 분석
- 푸시 알림
